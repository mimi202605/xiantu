# 世界·地图·NPC 综合设计档案

> 整合世界心跳、地图系统、NPC 人物系统（重点/普通）、NPC 对话、多维人设、社交流言、情感记忆、世界互动痕迹、专属小任务等模块。各模块相互依赖，需统一规划、分阶段实现。

---

## 一、设计原则与依赖关系

### 1.1 核心约束

| 约束 | 说明 |
|------|------|
| **AI 生成** | 所有新数据由 LLM 产出，需通过 tavern_commands 写入；避免复杂 schema 导致 LLM 迷失 |
| **Token 效率** | 选择性注入 prompt；非当前场景 NPC 只注入摘要；普通 NPC 仅在特定条件下注入 |
| **数据统一** | NPC 一律存于 `社交.关系`；重点/普通均存，通过 `类型` 字段区分 |
| **简单类型** | 禁止深层嵌套、动态 key 的复杂对象；优先 `string`、`number`、`string[]`、`Record<string, string>` |

### 1.2 模块依赖关系

```
地图系统 ──┬── 地点内部结构（内部：递归子地点，上级：父地点；父查子、子查父）
           ├── 探索记录（系统记录，用于 UI 迷雾）
           └── 地点 NPC 列表（与 社交.关系 对耦，仅存 NPC 名）

NPC 类型 ──┬── 重点 NPC：主线/关系表/人物关系 UI 全展示
           ├── 普通 NPC：仅在地点、提及、重要 NPC 关系网中出现
           └── 升级逻辑：普通 → 重点（玩家与其交流后）

世界心跳 ──┬── 依赖：地图（位置）、NPC 类型（更新优先级）
           └── 预留：更新队列、心跳日志、地图分步更新人物去向

NPC 对话 ──┬── 依赖：RelationshipNetworkPanel、重点 NPC 列表
           └── 触发升级：与普通 NPC 对话 → 升级为重点

NPC 真实感 ── 人设、情感记忆、流言、痕迹、小任务（后续阶段）
```

### 1.3 实现优先级

| 阶段 | 内容 | 说明 |
|------|------|------|
| **Phase 1** | 地图系统 + NPC 类型 | 先实现，预留后续接口 |
| **Phase 2** | NPC 对话系统 | 依赖 Phase 1 的重点 NPC 列表 |
| **Phase 3** | 世界心跳系统 | 依赖地图、NPC 位置、类型 |
| **Phase 4** | NPC 真实感（人设、情感记忆等） | 依赖 plan-npc-realism-phase.md |

## 二、地图系统

### 2.1 设计目标

- 地图地点根据剧情逐步生成，记录与显示，形成 RTS 迷雾感
- 在现有 `世界.信息.地点信息` 基础上增加**内部结构**（子地点）
- 地图 UI：放大可展开内部结构，缩小可隐藏

### 2.2 数据结构

**设计原则**：

- **LLM 生成**：结构尽量简单，字段单一语义，避免中英重复（如 `name`/`名称`、`description`/`描述` 二选一，统一用中文）
- **记录与显示**：生成后我们记录，用于 UI 显示；**探索记录**、**地点 NPC 列表**等**不每次发给 LLM**，**不每次更新**，不影响 token

#### 2.2.1 地点条目（存储递归，注入时提取）

设计参考 NPC 人物关系：`社交.关系` 为 `Record<名称, 档案>`，每个 NPC 有 `关系: Record<对方名, 关系标签>` 实现双向可查。地点同理：**存储**用递归结构，**父查子、子查父**均可；**发给 LLM 时**提取为扁平内部结构，避免复杂嵌套。

```ts
// 世界.信息.地点信息：Record<名称, LocationEntry> 或 LocationEntry[]（按名称索引）
// 与 NPC 关系类似：地点为独立条目，通过 名称 关联
interface LocationEntry {
  名称: string;           // 唯一标识，如 "百货大楼"、"百货大楼一楼"
  描述?: string;          // 简短描述，可选
  上级?: string;         // 父地点名称，根节点无；子查父
  内部?: LocationEntry[]; // 子地点，递归；父查子
}

// 坐标、已探索、已发现、迷雾：不纳入 LLM 生成结构
// - 坐标：可选，若有则用于地图 UI 定位；LLM 不强制生成
// - 迷雾：由 探索记录 推导
```

**双向查阅**：

| 查阅 | 方式 |
|------|------|
| 父查子 | `entry.内部` → 子地点数组 |
| 子查父 | `entry.上级` → 父地点名称，再于 `地点信息` 中按名称查找 |

**LLM 注入**：发给 LLM 时，将递归的 `内部` **提取为扁平结构**（如 `内部名称列表: string[]` 或一层嵌套），避免深层对象导致 token 膨胀与格式错误。提取逻辑由 `buildNarrativeState` 或等价模块实现。

**字段说明**：

| 字段 | 用途 | LLM 生成 |
|------|------|----------|
| 名称 | 唯一标识，与 `当前位置.描述` 匹配 | 是 |
| 描述 | 简短说明，可选 | 是 |
| 上级 | 父地点名称，子查父 | 是 |
| 内部 | 子地点数组，递归，父查子 | 是 |
| ~~坐标~~ | 地图 UI 定位，可选 | 否 |
| ~~已探索/已发现/迷雾~~ | 由 **探索记录** 推导 | — |

#### 2.2.2 探索记录（系统记录，不发给 LLM）

```ts
// 存储于：角色.探索记录 或 世界.状态.探索记录
探索记录?: string[];  // 已探索的地点 名称 列表
```

- **用途**：UI 迷雾显示（在记录中 = 已发现 = 非迷雾）
- **更新时机**：玩家首次进入某地点时 `push`，非每回合
- **注入 LLM**：不注入，或仅在「需要判断是否已探索」时注入极简摘要

#### 2.2.3 地点 NPC 列表（存于各地点内）

```ts
// 存于每个 LocationEntry 内
interface LocationEntry {
  名称: string;
  描述?: string;
  上级?: string;
  内部?: LocationEntry[];
  地点NPC?: string[];  // 本地的 NPC 名列表
}
```

- **对耦**：只存 NPC **名**（`string[]`），不存完整档案；完整数据在 `社交.关系`
- **存于地点内**：可追溯各地点的 NPC；玩家离开后**重点 NPC 会自行离开**，**普通 NPC 留守**
- **用途**：UI 显示某地点的 NPC；玩家处于某地点时，按此表注入该地点 NPC 给 LLM
- **更新时机**：玩家进入地点时由 LLM 或系统生成/更新，非每回合

#### 2.2.4 与现有 WorldLocation 的兼容

现有 `WorldLocation` 含 `名称`、`描述`、`coordinates`、`类型` 等。新建地点时：

- 使用 `LocationEntry` 结构：`{ 名称, 描述?, 上级?, 内部? }`，`内部` 递归
- `coordinates`、`类型` 等可选，由系统或后续补全
- 旧存档的 `WorldLocation` 保持不变，迁移时可选补全 `上级`、`内部`

### 2.3 地点生成规则

- **进入地点时**：若地点不存在，由 LLM 通过 tavern_commands 生成并写入；顶层地点 `push` 到 `世界.信息.地点信息`，或按现有结构合并
- **进入子地点时**：若父地点的 `内部` 中无该子地点，`push` 到父地点的 `内部` 数组；子地点需设 `上级` 为父地点名称
- **结构**：描述可简短；`内部` 递归，支持多层

### 2.4 地图 UI 预留

- 放大：展开 `内部`，递归显示子地点
- 缩小：折叠，仅显示当前层
- 迷雾：`探索记录` 包含该地点名称 → 已发现；否则灰显

### 2.5 与 NPC 的耦合与解耦

- **位置解耦**：NPC 的 `当前位置.描述` 使用**地点名称**（如 "百货大楼一楼"），不绑定坐标
- **访问地点时注入**：玩家处于某地点时，从 `地点NPC[地点名称]` 取 NPC 名列表，再从 `社交.关系` 取对应摘要注入 LLM（见 3.4）
- **接口**：`getNpcsAtLocation(地点名称): string[]`，数据来源为 `世界.信息.地点NPC[地点名称]` 或从 `社交.关系` 按 `当前位置.描述` 匹配

## 三、NPC 类型（重点 / 普通）

### 3.1 类型定义

在 `NpcProfile` 中新增**必填**字段：

```ts
类型: "重点" | "普通";  // 必填，新建 NPC 时由 LLM 或规则设定
```

- **重点 NPC**：与玩家有交集、故事中提及、或玩家主动交流过
- **普通 NPC**：系统被动生成的背景 NPC，与玩家无交集

### 3.2 存储与展示

| 维度 | 规则 |
|------|------|
| **存储** | 重点、普通**均**存于 `社交.关系`，key 为 NPC 名 |
| **游戏变量·存档数据** | 展示 `社交.关系` 时**排序**：重点 NPC 在前，普通 NPC 在后；同类型内可按好感度/名字等二次排序 |
| **人物关系 UI** | **仅展示重点 NPC**；普通 NPC 不出现在 RelationshipNetworkPanel 的人物列表中 |
| **关系表（实体与关系）** | **仅包含重点 NPC**；`deriveFrom社交关系` 需过滤，只输出 `类型==="重点"` 的 NPC 作为实体和边 |
| **重要 NPC 的关系网** | 重点 NPC 的 `关系` 字段中，**可以**引用普通 NPC（`关系: { "某普通NPC": "路人" }`）；在 NPC 详情页展示该 NPC 的 `关系` 时，可显示「B: 路人」等文本，但普通 NPC **不**作为关系表/实体图中的节点；全局关系表仅含重点 NPC |

### 3.3 严谨的过滤逻辑

**定义**：

- `isImportantNpc(npc: NpcProfile): boolean` → `npc.类型 === "重点"`
- `isOrdinaryNpc(npc: NpcProfile): boolean` → `npc.类型 === "普通"`

**各模块使用规则**：

| 模块 | 数据源 | 过滤规则 |
|------|--------|----------|
| **RelationshipNetworkPanel 人物列表** | `社交.关系` | 仅 `类型==="重点"` |
| **deriveFrom社交关系（关系表/实体图）** | `社交.关系` | 实体：仅 `类型==="重点"`；边：仅当 fromId 与 toId 均为重点 NPC 时输出（普通 NPC 不作为节点，故涉及普通 NPC 的边也不输出） |
| **GameVariablePanel 社交.关系 展示** | `社交.关系` | 全部展示，但**排序**：重点在前，普通在后 |
| **buildNarrativeState / memoryRetrieval** | `社交.关系` | 见 3.4 |
| **AIBidirectionalSystem 主回合** | 同上 | 见 3.4 |

### 3.4 LLM 注入规则（核心）

普通 NPC **不应**出现在大多数发给 LLM 的逻辑中，**仅在**以下情况注入：

1. **玩家处于该 NPC 所在地点**：`角色.位置.描述` 与 `NPC.当前位置.描述` 匹配（或同属一地点）时，将该地点的**所有 NPC**（含普通）注入当前场景
2. **被提及**：主故事/指令中显式提到某 NPC 名，则注入该 NPC
3. **在重点 NPC 的关系网内**：若重点 NPC A 的 `关系[B]` 存在且 B 是普通 NPC，当 A 被注入时，B 可作为「A 认识的人」以**摘要形式**（如 "A 认识 B（路人）"）注入，**不**注入 B 的完整档案

**其他情况**：普通 NPC **不**进入 `# 相关实体`、`# 关系`、`recentNpcNames`、`实时关注` 等逻辑。

### 3.5 升级逻辑：普通 → 重点

**触发条件**：玩家与普通 NPC 发生**有意义的交流**，包括但不限于：

- 主线剧情中与该 NPC 对话/互动
- 通过 **NPC 对话系统**（RelationshipNetworkPanel 对话 tab）与该 NPC 对话
- 其他系统判定为「产生交集」的行为

**升级操作**：

```
set 社交.关系.{NPC名}.类型 "重点"
```

- 实现位置：`processGmResponse` 后扫描 tavern_commands，若存在与某 NPC 的互动指令且该 NPC 当前为普通，则追加 `set 社交.关系.{NPC}.类型` `"重点"`
- 或：NPC 对话系统在「结束对话」时，若对话对象为普通 NPC，则生成的更新指令中**必须**包含 `set 社交.关系.{NPC}.类型` `"重点"`

### 3.6 新建 NPC 时的类型设定

- **主线/剧情生成**：由 LLM 在 `set 社交.关系.{NPC}` 时填写 `类型`；若未填，**默认** `"重点"`（故事中出现的视为重点）
- **地图/地点生成**：系统或 LLM 在生成某地点的背景 NPC 时，填写 `类型: "普通"`
- **迁移**：旧存档无 `类型` 字段时，视为 `"重点"`（兼容已有 NPC）

## 四、NPC 对话系统

### 4.1 入口与布局

- 在 **RelationshipNetworkPanel** 的详情区，与「基本信息」「实时状态」「记忆档案」「背包」「原始数据」并列，新增 **「对话」tab**
- 仅当 `selectedPerson` 存在且为**重点 NPC** 时，该 tab 可用（普通 NPC 不在人物列表中，无法直接选到；若通过其他入口与普通 NPC 对话，对话完成后会升级为重点，此时会出现在列表中）
- 注：若后续支持「从地图进入某地点 → 选择该地点的普通 NPC 对话」，则需在对话完成时执行升级逻辑

### 4.2 API 与调用方式

- **默认**：复用主线 AI 服务（与主故事相同的 API）
- **可定制**：在 API 管理 → 功能分配中，新增 `npc_dialogue` 类型，可为 NPC 对话分配独立 API（类似 `memory_summary`、`event_generation` 等辅助功能）
- 调用时使用 `apiManagementStore` 中 `npc_dialogue` 对应的 API 配置

### 4.3 Prompt 设计

- **与主故事相同**：角色卡、世界状态、记忆等结构一致
- **额外强调**（在 system 或 context 中）：
  - 这是**玩家与 [NPC名] 之间的私人对话**，不推动主线剧情
  - 对话内容应围绕两人交流，不涉及重大剧情推进
  - 输出格式：`{ "content": "对话正文", "tavern_commands": [...] }`，与主故事一致

### 4.4 结束对话时的处理

1. **总结**：将对话内容总结为 **memory takeaway**（1–3 条简短摘要）
2. **生成更新指令**：`tavern_commands` 中**仅**允许以下类型的更新：
   - `push 社交.关系.{NPC}.记忆`（NPC 记忆）
   - `add 社交.关系.{NPC}.好感度`（可选）
   - `set 社交.关系.{NPC}.当前内心想法`（可选）
   - 玩家记忆：若存在 `社交.记忆` 的写入路径，可允许
   - **禁止**：修改主线相关状态、世界事件、其他 NPC 的档案、角色位置等

3. **引导**：在 prompt 中明确写出「指令必须只更新小部分数据，不得影响主剧情」

### 4.5 分步生成

- 若使用分步生成（step1/step2），step2 的 `tavern_commands` 同样遵守 4.4 的约束
- 校验：`executeCommand` 或后处理中，对 NPC 对话产生的 commands 做**白名单校验**，拒绝超出范围的修改

在 Phase 1 实现地图与 NPC 类型时，预留以下接口：

```ts
// 更新优先级：供心跳调度使用
getNpcUpdatePriority(saveData: SaveData, worldEvent?: GameEvent): string[];
```
