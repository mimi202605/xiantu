# 世界心跳/呼吸系统 — 实现计划（v2）

基于 `docs/design note` 第 21–27 条、代码库调研及你的确认整理。以下为定稿设计。

---

## 一、需求与定稿决策摘要

| 项 | 决策 |
|----|------|
| 周期 | 仅「每 N 回合」；需增加全局回合序号并落盘。 |
| 锁定 | 在每个 NPC 上增加 `心跳锁定?: boolean`，便于关系面板 toggle。 |
| 关联性 | 首版使用简化规则：类型(重点)+实时关注+与玩家关系+事件相关人物。 |
| 在做事项 | 新增 NPC 字段「在做事项」；主回合生成/更新（简短，类似内心想法）；心跳一并更新 位置/状态/内心想法/在做事项。关系面板新增 Tab、改 prompt 与数据结构。 |
| 心跳 UI | 与世界事件类似：列表列出每次心跳；每条可展开查看每位 NPC 的更新情况与功能按键。历史保存 N 条，N 在设置中可配置。 |
| 事件触发范围 | 事件.相关人物 + 主角关系网 + 关注 NPC；不更新仅普通且与事件/主角无关的 NPC。 |
| 遗忘机制 | 每个 NPC 增加「被主游戏更新的回合序号」；若 NPC 长时间未被主游戏更新则不再参与心跳更新。设置中增加「遗忘回合数」。 |
| 设置 | 世界心跳相关游戏设置放在一起并做分组（group）。 |

---

## 二、数据结构（定稿）

### 2.1 全局回合序号

- 存于 `元数据.回合序号` 或 `系统.状态.回合序号`（建议 `元数据.回合序号`，与时间同级）。
- 主回合成功完成并写回存档时 +1；开局、读档时若缺失则置 0 或 1。

### 2.2 世界.状态.心跳

```ts
世界: {
  信息: WorldInfo;
  状态: {
    探索记录: string[];
    心跳?: {
      启用: boolean;
      周期数值: number;              // 每 N 回合
      上次心跳回合序号?: number;
      历史条数: number;              // 保留最近 N 条，设置可配置，默认如 20
      遗忘回合数: number;            // 超过此回合数未被主游戏更新的 NPC 不再参与心跳，默认如 30
      历史?: HeartbeatRecord[];
    };
  };
}
```

- 不再使用「周期类型」「现实分钟」「锁定NPC 名单」；锁定仅用 NpcProfile.心跳锁定。

### 2.3 NpcProfile 扩展

```ts
// 实时状态（与 当前外貌状态、当前内心想法 同级）
在做事项?: string;           // 简短，与内心想法类似，如 "在客栈打杂"

// 世界心跳与主回合追踪
心跳锁定?: boolean;          // true 则不参与心跳更新
上次主回合更新回合?: number; // 最近一次被**主回合**（主游戏）更新时的 元数据.回合序号；仅主回合更新时写入，心跳更新不写入
```

- **重要**：`上次主回合更新回合` 必须**仅**在「主回合」更新 NPC 时写入。心跳同样通过 tavern_commands 更新 NPC，因此不能笼统地按「应用了 tavern_commands 就写」——需**区分来源**：
  - **主回合**：在应用**主回合**返回的 tavern_commands 之后，由主回合流程单独做一步：根据本批已应用的、且 key 以 `社交.关系.` 开头的命令，解析出被更新的 NPC 名字，将这些 NPC 的 `上次主回合更新回合` 设为当前 `元数据.回合序号`。
  - **心跳**：心跳有独立的「应用命令」入口（或调用同一 apply 函数但不触发「标记主回合更新」）；该路径**绝不**写入 `上次主回合更新回合`。
- 实现方式二选一即可：(A) 主回合在 apply 之后显式根据已应用的命令收集 NPC 名并写回 上次主回合更新回合；(B) 若存在共用的 apply 函数，为其增加参数如 `markAsMainTurnUpdate?: boolean`，仅主回合调用时传 true，内部仅在 true 时写 上次主回合更新回合。推荐 (A)，主回合逻辑自洽、不污染共用层。
- 心跳候选筛选：排除 `心跳锁定 === true`，且排除 `当前回合序号 - (npc.上次主回合更新回合 ?? 0) > 遗忘回合数` 的 NPC。

### 2.4 单次心跳记录（可展开 UI）

```ts
interface HeartbeatRecord {
  时间: string;              // 游戏时间或 ISO
  回合序号: number;
  触发方式: '周期' | '事件' | '手动';
  相关事件ID?: string;
  更新列表: {
    npc名字: string;
    更新摘要?: string;        // 如 "位置、在做事项、内心想法"
    更新前?: Partial<NpcProfile>;  // 可选：用于单条回溯与展开展示
    更新后?: Partial<NpcProfile>;  // 可选：用于展开展示
  }[];
  快照: Record<string, NpcProfile>;  // 更新前的完整 NPC，用于全部/单条回溯
}
```

- 历史条数由 `世界.状态.心跳.历史条数` 控制，只保留最近 N 条。

### 2.5 设置中的「世界心跳」分组

- 配置项（均写入 世界.状态.心跳）：
  - 启用世界心跳
  - 更新周期（每 N 回合）
  - 历史保留条数（N 条）
  - 遗忘回合数（超过此回合数未被主游戏更新的 NPC 不再参与心跳更新）
- 上述放在设置面板的同一分组（如「世界心跳」），与世界事件分组并列或放在「游戏功能」下的一组。

## 三、「在做事项」串联修改清单

1. **类型**（`types/game.d.ts`）  
   - `NpcProfile` 增加 `在做事项?: string`。

2. **数据修复与迁移**（`dataRepair.ts` / `saveMigration.ts`）  
   - 旧存档 NPC 缺失时补 `在做事项: ''` 或 `'未知'`（与 当前内心想法 一致）。

3. **Prompt 与数据定义**  
   - `dataDefinitionsMing.ts`（及通用版如需要）：在关系/NPC 结构中增加「在做事项」说明；set 示例 `社交.关系.[NPC名].在做事项`。  
   - 主回合相关 prompt（如 `coreRulesMing.ts`、`cotCore.ts`、`inlinePromptsMing.ts`）：要求主回合中若涉及某 NPC，可输出 set 该 NPC 的 `当前外貌状态`、`当前内心想法`、`在做事项`（简短一句）；与内心想法同级别、同长度约束。  
   - 心跳专用 prompt（新建）：输入世界/时间/事件摘要与候选 NPC 当前状态，输出仅针对这些 NPC 的 set（位置、当前外貌状态、当前内心想法、在做事项）。

4. **主回合执行层**（`AIBidirectionalSystem.ts`）  
   - **仅在主回合**：在应用**主回合**返回的 tavern_commands 之后（即主回合专用代码路径，而非心跳等其它入口），根据本批已应用的命令中 key 以 `社交.关系.` 开头的项解析出被更新的 NPC 名字，将这些 NPC 的 `上次主回合更新回合` 设为当前 `元数据.回合序号`。心跳应用的 tavern_commands 走另一路径，**不得**写入 `上次主回合更新回合`。

5. **关系面板**  
   - 在人物详情中新增 Tab「在做事项」（与「基本信息」「记忆」等并列），展示并可编辑（若你希望可编辑）`在做事项` 字段；展示逻辑与「当前内心想法」类似。

6. **心跳逻辑**  
   - 心跳更新时，将「在做事项」与位置、状态、内心想法一起作为可更新字段，由心跳 prompt 输出并执行 set。

## 四、实现阶段划分（按依赖顺序）

### Phase 1：回合序号 + 心跳与 NPC 数据结构

1. **回合序号**  
   - `元数据.回合序号` 类型与默认值；`loadFromSaveData` / `toSaveData` 同步；主回合成功结束时在写回存档前递增（在 `AIBidirectionalSystem` 或调用其的流程中）。

2. **NpcProfile 扩展**  
   - 增加 `在做事项?`、`心跳锁定?`、`上次主回合更新回合?`；dataRepair/saveMigration 与 dataValidation 中补默认值。

3. **世界.状态.心跳 类型与默认**  
   - 类型定义；`心跳.历史条数`、`遗忘回合数` 的默认值（如 20、30）；dataRepair/saveMigration 中为旧存档写入默认心跳配置（启用=false）。

4. **Store**  
   - `gameStateStore` 读写 `世界.状态.心跳`、`元数据.回合序号`；`toSaveData` 写回。

### Phase 2：「在做事项」在主回合与关系面板的落地

5. **Prompt 与数据定义**  
   - 在数据定义中增加「在做事项」字段与 set 示例；在主回合规则/CoT/内联提示中要求对涉及 NPC 可更新 当前外貌状态/当前内心想法/在做事项（简短）。

6. **主回合写回时更新「上次主回合更新回合」**  
   - 仅在**主回合**应用 tavern_commands 的代码路径中，在 apply 完成后根据本批命令解析出被更新的 NPC（key 以 `社交.关系.` 开头），写回 `社交.关系[npc].上次主回合更新回合 = 元数据.回合序号`。心跳应用的命令不执行此逻辑。

7. **关系面板**  
   - 人物详情新增 Tab「在做事项」，展示并可选编辑 `在做事项`；人物卡片或详情中增加「锁定心跳更新」toggle，绑定 `心跳锁定`。

### Phase 3：心跳核心逻辑

8. **候选选择**  
   - 排除 `心跳锁定 === true`；排除「当前回合序号 - (上次主回合更新回合 ?? 0) > 遗忘回合数」；周期模式用简化规则（重点+关注+与玩家关系）；事件模式：事件.相关人物 + 主角关系 + 关注 NPC，且不更新仅普通且无关的 NPC。

9. **心跳专用 Prompt + 调用**  
   - 输入：世界/时间/近期事件摘要、候选 NPC 列表及其当前 位置/状态/内心想法/在做事项/关系；输出：仅 set 社交.关系.{名字} 的 位置、当前外貌状态、当前内心想法、在做事项（tavern_commands）。

10. **单次心跳执行**  
    - 对候选 NPC 调用 AI → 执行命令前对这批 NPC 做快照 → 执行命令 → 同步 Store、`calibrateNpcLocationSync` → 追加 HeartbeatRecord（含 更新列表 与 快照），截断历史至 `历史条数`。

11. **触发入口**  
    - 周期：主回合成功结束后检查 启用 + (当前回合序号 - 上次心跳回合序号 >= 周期数值)，满足则执行心跳并更新 上次心跳回合序号。  
    - 事件：`maybeTriggerScheduledWorldEvent` 在 push 事件后若启用心跳则执行一次事件驱动心跳（传入该事件）。  
    - 手动：UI 按钮触发，用周期模式候选。

### Phase 4：心跳 UI 与设置分组

12. **世界心跳 UI（与世界信息/事件类似）**  
    - 列表：每条心跳一行，显示 时间、回合序号、触发方式、更新 NPC 数量；每条可展开，展开后显示每位 NPC 的更新情况（更新摘要或 更新前/更新后 关键字段）及「单条回溯」按钮。  
    - 顶部或工具栏：手动触发、全部回溯（最近一条）、可配置历史条数说明（来自设置）。  
    - 全部回溯：用最近一条的 快照 写回 社交.关系，同步 Store + 地点校准，并从历史中移除该条。

13. **设置面板**  
    - 新增「世界心跳」分组（与「世界事件」等并列或同属「游戏功能」）：  
      - 启用世界心跳  
      - 每 N 回合更新（数字输入）  
      - 历史保留条数（N 条）  
      - 遗忘回合数（超过则不再参与心跳更新）  
    - 保存到 世界.状态.心跳，并同步 Store。

### Phase 5（可选）后续

- 关联性细化、地图分步更新人物去向、NPC 间关系与记忆在世界心跳中的更新等。

## 六、小结

- **周期**：每 N 回合，依赖 元数据.回合序号。  
- **锁定**：NpcProfile.心跳锁定，关系面板 toggle。  
- **在做事项**：新字段 + 主回合生成/更新 + 关系面板新 Tab + 心跳更新。  
- **遗忘**：NpcProfile.上次主回合更新回合 + 设置「遗忘回合数」，超时则不再参与心跳。  
- **UI**：心跳列表可展开、每条展示每名 NPC 的更新与单条回溯；历史条数可配置。  
- **设置**：世界心跳相关项成组配置。

按 Phase 1→2→3→4 实现即可；Phase 5 为后续增强。
